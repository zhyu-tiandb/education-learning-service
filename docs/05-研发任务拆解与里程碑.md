# 教育学习助手系统研发任务拆解与里程碑

## 1. 研发原则

本项目必须按“小闭环、可验收、可回退”的方式推进。

不允许一开始同时做前端、视频、知识图谱、智能问答等大范围能力。V1先跑通：

```text
教材图片 → OCR → 知识点 → 错题 → 掌握度 → 报告
```

## 2. 里程碑规划

## M1：基础工程与本地存储

周期建议：1周

目标：

1. FastAPI工程搭建。
2. SQLite初始化。
3. 本地目录初始化。
4. 文件上传能力。
5. 任务状态机。

输出物：

```text
后端工程骨架
数据库初始化脚本
本地目录结构
健康检查接口
上传接口
任务状态查询接口
```

验收标准：

1. 服务可以启动。
2. 上传图片能保存到 data/raw。
3. 任务状态能记录到数据库。
4. 日志能正常写入。

## M2：教材OCR与知识点提取

周期建议：2周

目标：

1. 教材图片OCR识别。
2. OCR结果保存。
3. 知识点提取。
4. Markdown知识文档生成。

输入：

```text
教材图片
教材元数据
```

输出：

```text
OCR文本
知识点JSON
教材知识Markdown
SQLite记录
```

验收标准：

1. 上传一张教材图片后能生成OCR文本。
2. 能提取章节、知识点、例题、易错点。
3. 能保存到 data/knowledge/textbook_knowledge。
4. 低置信度内容能标记。

## M3：作业/错题识别与分析

周期建议：2周

目标：

1. 上传作业/错题图片。
2. 识别题目和学生答案。
3. 匹配知识点。
4. 生成错因分析。

输出：

```text
错题记录
知识点关联
错因标签
订正建议
```

验收标准：

1. 能识别清晰作业图片。
2. 能输出至少一个相关知识点。
3. 能输出错因类型。
4. 能保存错题记录。
5. 支持人工修正错因标签。

## M4：掌握度评估与学习报告

周期建议：1-2周

目标：

1. 根据错题更新知识点掌握度。
2. 聚合薄弱知识点。
3. 生成Markdown学习报告。
4. 输出学习方案。

输出：

```text
掌握度记录
学习报告
学习方案
```

验收标准：

1. 能按知识点生成掌握度分数。
2. 能识别薄弱知识点。
3. 能生成Markdown报告。
4. 报告内容可读、可复制、可归档。

## M5：简单前端或管理页面

周期建议：2周

目标：

1. 提供最小页面。
2. 支持上传教材。
3. 支持上传作业。
4. 查看分析结果。
5. 查看报告。

输出：

```text
简单Web页面
上传页面
报告页面
```

验收标准：

1. 家长不用命令行即可完成一次分析。
2. 页面能展示报告。
3. 页面能查看错题列表。

## 3. 任务拆解

### T1 工程初始化

负责人：后端  
依赖：无  
输出：FastAPI工程、目录结构、配置文件  
验收：服务可启动，健康检查通过。

### T2 数据库设计与初始化

负责人：后端  
依赖：T1  
输出：SQLite表结构、初始化脚本  
验收：数据库可创建，表结构完整。

### T3 文件上传模块

负责人：后端  
依赖：T1  
输出：上传接口、文件保存逻辑  
验收：图片上传后可在本地目录找到。

### T4 OCR模块

负责人：AI/后端  
依赖：T3  
输出：OCR服务封装  
验收：图片可输出文本和置信度。

### T5 教材知识点提取模块

负责人：AI/后端  
依赖：T4  
输出：知识点JSON、Markdown文档  
验收：能从教材文本中提取知识点。

### T6 作业分析模块

负责人：AI/后端  
依赖：T4、T5  
输出：题目识别、错因分析、知识点匹配  
验收：能生成错题分析记录。

### T7 掌握度模块

负责人：后端  
依赖：T6  
输出：掌握度计算规则和记录  
验收：错题分析后能更新知识点分数。

### T8 报告生成模块

负责人：后端/AI  
依赖：T6、T7  
输出：Markdown报告  
验收：能生成一份完整学习报告。

### T9 简单前端页面

负责人：前端  
依赖：T3、T8  
输出：上传和报告查看页面  
验收：可以完成一次端到端流程。

## 4. 关键评审点

### R1 需求评审

在开发前完成，重点确认：

1. V1范围是否收敛。
2. 是否明确不做课堂视频。
3. 是否明确不做完整知识图谱。
4. 是否明确试点学科。
5. 是否明确验收样例。

### R2 技术方案评审

重点确认：

1. OCR方案是否可运行。
2. LLM调用是否可配置。
3. 数据是否本地落盘。
4. 任务状态是否可追踪。
5. 错误日志是否完整。

### R3 联调评审

重点确认：

1. 上传接口与处理任务是否打通。
2. OCR与知识提取是否打通。
3. 错题分析与掌握度是否打通。
4. 报告是否能读取真实数据。

### R4 验收评审

重点确认：

1. 是否跑通端到端流程。
2. 是否有本地数据文件。
3. 是否有数据库记录。
4. 是否有报告输出。
5. 是否有失败日志和人工修正入口。

## 5. 项目风险

| 风险 | 表现 | 应对 |
|---|---|---|
| 范围膨胀 | 同时做视频、图谱、问答 | V1严格只做教材+错题+报告 |
| OCR不稳定 | 识别错误较多 | 增加置信度和人工修正 |
| LLM误判 | 错因分析不可靠 | 输出证据和置信度 |
| 前端拖慢进度 | 页面耗费过多时间 | V1先API验证 |
| 数据不可追溯 | 不知道分析依据 | 保存原图、OCR、JSON、报告 |
